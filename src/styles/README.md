# src/styles

| Path                  | What it is                                                                                                                   |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| `portfolio/index.css`  | Entry point for all portfolio visual styling. Imported once, by `components/layout/PortfolioPage.tsx`.                        |
| `portfolio/*.css`      | The same styling, split into ordered partials (design tokens, per-section base layout, and each historical "color world" override pass). |

`src/styles.css` (one level up, **not** in this folder) is a different
file — the Tailwind/shadcn base stylesheet generated by the project
scaffold, unrelated to this site's actual design.

## Why this is split into numbered partials instead of true per-section files

A lot of this site's per-section styling was authored in more than one
pass: a base layout block, and then one or more later "color world"
revisions that override the same selectors with a higher-specificity or
later cascade position. **Later rules intentionally win.** A naive
per-section split (e.g. one `about.css`, one `projects.css`) would
silently change that load order — and therefore the final colors —
because a single selector's rules are currently spread across multiple
non-contiguous blocks in the original source.

Instead, `portfolio/index.css` `@import`s 23 partials in the **exact
original source order**, split at the file's existing top-level comment
boundaries (e.g. `06-motion-upgrade-section-identities.css`,
`07-palette-patch-v2.css`, `08-paper-cut-color-worlds-v3.css` are three
successive full-site revisions, each superseding the last). This was
verified to reconstruct the original file byte-for-byte, so the rendered
design is unchanged.

**Import order in `index.css` is load-bearing — do not reorder it.**

### Recommended follow-up (not done here, to stay zero-risk)

The real fix for the underlying messiness is to pick a single "final"
value per selector and delete the superseded layers (e.g. fold
`07-palette-patch-v2.css` and `08-paper-cut-color-worlds-v3.css` into
their section's base theme file, since `08` supersedes `07`). That's a
visual-diffing task best done with a live preview/screenshot pass per
section, so it's deliberately left as a follow-up rather than risked in
this structural reorganization.
